# 实时频谱选频与循环回放解码前端实现记录

> 2026-07-15 后续修正：本文记录的是第一版单载波诊断 UI。当前 1x、多载波、自适应
> 噪声底、异步连续解码与 TETRA 提速实现见
> [2.5 MHz 一倍速多载波并行前端修正与验收记录](../validation/2.5MHz一倍速多载波并行前端修正与验收记录.md)；若两者描述冲突，以后者为准。

## 1. 结论与当前边界

截至 2026-07-15，项目已具备一个可运行的单载频离线实时前端：对 BVSP、无头交错 IQ
或双通道 IQ WAV 持续绘制功率谱和瀑布图，用户点击载波后，从文件开头重新回放，经
有状态 DDC 变为 120 kS/s 基带，再进入 DMR、P25、dPMR、NXDN、TETRA 制式竞赛、
Epoch 管理、赢家连续解码和 PDU 展示。

入口为：

```matlab
radio_frontend
```

本阶段是功能正确性和交互流程基线，并非 61.44 MS/s 硬实时承诺。一次只处理一个用户
选中的载频；不使用 PFB，不自动创建大量候选，也不包含尚未确定的 SDR 驱动和推送队列。

## 2. 已实现的数据流

```text
BVSP / RAW / stereo WAV
        |
        v
FileLoopSource（10 ms拉取，全局样点号单调）
        |
        +-----------------------> SpectrumEngine
        |                          65536点FFT / Average / Max Hold
        |                          有界single瀑布图，不保存宽带IQ历史
        |
        v
已选载频 DDC（NCO + 抗混叠滤波 + 抽取）
        |
        | 120 kS/s、10 ms连续块
        v
BasebandMicroBatcher（默认100 ms）
        |
        v
ActivityDetector / RF Epoch
        |
        v
DMR | P25 | dPMR | NXDN | TETRA Probe Race
        |
        v
Winner Catch-up -> LOCKED -> 持续解码 -> PDU表/JSON
```

文件源采用拉取方式。当前块全部处理完才读下一块，因此离线回放处理慢时只会使墙钟
回放变慢，不会无限堆积内存，也不会丢失文件样点。未来 SDR 源是推送式，不能依赖这种
自然背压，仍需有界队列和明确的溢出策略。

## 3. 代码结构

### 顶层前端

- `radio_frontend.m`：UI、timer 调度、状态展示和程序化控制；
- `carrier_scope.m`：继续保留为静态频谱检查器，不承担流式生命周期。

### 循环文件源 `+radio/+replay`

- `fileLoopSourceInit`：识别 BVSP/RAW/WAV 并建立 10 ms 源；
- `fileLoopSourceRead`：返回全局时间轴上的 `IqChunk`；
- `fileLoopSourceReset`：回到文件开头并重置逻辑时间；
- `fileLoopSourceClose`：幂等关闭文件。

三种模式：

| 模式 | EOF 行为 | 用途 |
|---|---|---|
| `once` | 结束并 finalize | 正常离线验收 |
| `continuous-test` | 无间断地接到下一遍，全局样点继续 | 短样例扩展分类观察窗 |
| `epoch-repeat` | 插入静默，再用 discontinuity 开始下一遍 | 验证独立 Epoch 上报 |

`continuous-test` 的文件接缝不是物理连续信号，输出明确保留
`syntheticContinuousReplay=true` 标志，不能用来证明相位连续性能。

### 频谱引擎 `+radio/+scope`

- `spectrumInit/spectrumFeed/spectrumSnapshot`：增量 FFT、线性功率平均、Max Hold 和
  环形瀑布图；
- `resetMaxHold`：只清除峰值保持；
- `refineCarrier`：在点击点附近做信道能量搜索和功率质心精调。

默认 NFFT 为 65536；61.44 MS/s 时频率栅格为 937.5 Hz。瀑布图默认最多 200 行、
最多 4096 个显示频点，历史宽带 IQ 不会为绘图保留。

### 流式选频链 `+radio/+tuned`

- `streamScannerInit`：建立/预热 DDC、进程池和 `RaceCoordinator`；
- `streamScannerFeed`：逐块下变频并路由；
- `streamScannerQueueBaseband`：把 10 ms DDC 输出合并为检测微批；
- `streamScannerCoordinatorFeed`：统一收集事件、Epoch 和带选频信息的 PDU；
- `streamScannerFinalize`：冲洗 DDC、送出不足一批的尾部、等待异步任务、关闭 Epoch。

DDC 预热使用固定零块，随后 `reset`，所以预热不会推进业务样点号，也不会制造虚假
Epoch。每个 PDU 的 `extra.tuned` 记录输入中心、目标频点、相对频偏、输入/输出采样率和
抽取倍数。

## 4. 两个真实数据调试结论

### 4.1 10 ms 活动检测会漏掉 DMR 短突发

最初版本把每个 10 ms DDC 输出直接送给活动检测。`DMR_signal/1.bvsp` 的目标信道功率
呈现约 40 ms 有信号、20 ms 静默的周期，而默认 `minOnSec=50 ms` 要求连续越过开启
门限。结果每次尚未累计到 50 ms 就被静默块清零，表现为频谱能看到载波，但识别始终
无 Epoch、无赢家。

离线稳定基线原本使用 `streamConfig.chunkDurationSec=100 ms`。最终实现没有降低门限，
而是在 DDC 与 Coordinator 之间恢复同样的 100 ms 微批语义：

```text
10 x 10 ms连续DDC块 -> 1 x 100 ms活动/竞赛块
```

这样既保留 DDC 的连续滤波状态和前端响应速度，又不会把 TDMA 时隙空隙误认为一次未
成立的载波。微批器同时处理非整尾块、样点间断、丢失计数和 discontinuity 传播。

### 4.2 EOF 不能立即取消异步 catch-up

进程池模式下，Probe 确认 DMR 后会异步启动赢家 catch-up。最初 EOF 立即送入关闭 Epoch
所需的静默块，速度远快于 worker 解码，导致赢家已经显示为 DMR，但 catch-up future
被 `signal_ended` 取消，PDU 数为零。

现在正常 EOF 的 finalize 顺序为：

1. 冲洗 DDC 和不足 100 ms 的基带尾部；
2. 用零长度、不推进逻辑时间的块轮询当前 Probe/catch-up；
3. 任务完成后再送入 `offHangSec` 静默关闭 Epoch；
4. 若达到任务等待超时，报告 `finalizeTaskWaitTimedOut=true`，随后显式取消；
5. 用户手工 Stop 不等待，直接采用快速取消，避免界面长时间无响应。

## 5. 用户操作

```matlab
cd('/home/lzkj/lzkj_workspace/matlab_docs')
startup
radio_frontend
```

操作顺序：打开文件 → Preview → 点击谱峰 → 选择精调带宽 → Run identification。

- BVSP 的采样率和绝对 RF 中心从头部读取；
- RAW 必须给出采样率；RF 中心可为 0，此时只显示相对频偏；
- `Protocols=all` 表示五制式竞赛；也可填写 `dmr,p25` 等子集；
- 解码运行期间不能换频；先 Stop，再点击新频点；
- `Save PDU JSON` 使用项目统一的 `radio.writeJson` 输出格式。

对于约一秒的样例，若目标制式需要长于文件的 Probe 窗口，应选 `continuous-test` 并把
循环数设到足够大。TETRA 最大观察窗为 6 秒，建议至少提供 6 秒逻辑数据并留裕量。

## 7. 当前限制与下一阶段接口

1. 只有一个活动选频链；换频会销毁旧 DDC 和 Coordinator。
2. 频谱供人选频，不做自动候选发现；这是当前避免 PFB 杂散候选和算力失控的稳定方案。
3. 61.44 MS/s MATLAB CPU DDC 加 FFT/GUI 不保证 `1x`；界面实时因子大于 1 表示处理
   比信号逻辑时间慢。
4. 文件源没有生产者/消费者队列，因为拉取本身提供自然背压；真实 SDR 不能沿用这个
   假设。
5. 基带 PDU 到宽带绝对样点的映射目前为按抽取倍数换算的近似值，未补偿滤波群时延，
   `mapping_includes_filter_delay=false`。
6. UI 当前没有单独的去重开关、置信度栏和完整运行报告导出；核心报告和事件仍保留在
   Scanner/Coordinator 状态中。

真实设备确定后，新增 `SdrSource` 应继续输出现有 `IqChunk`，并在其前后补充：固定容量
CI16 队列、硬件时间戳、丢块计数、溢出策略以及下一块的 `discontinuity=true`。频谱、
DDC、微批、Epoch 和五制式竞赛不应因设备型号而重写。
