# 2.5 MHz 五路近同步信号合成与验收

## 1. 目的与边界

`tools/synthesizeFiveSignal2p5MHz.m` 生成一个本地离线压力样例，用于验证当前
`radio_live_frontend` 在 2.5 MS/s 输入下同时处理五个已知载频的行为。它不是自然空口
录音，也不用于评价接收机灵敏度；它专门制造五路近同步到达，从而暴露 DDC、共享
worker 池、解码积压和最终收尾问题。

生成器只使用 MATLAB。Python 旧工程只作为 DMR/P25/dPMR/TETRA 原始录音的默认数据
目录，不参与合成或解码运行；NXDN 源来自当前工程的 `signal_data`。输出及 JSON 清单均
位于被 Git 忽略的 `signal_data`，不会进入仓库。

## 2. 默认场景

| 通道 | 制式 | 频偏 | 输出起点 | 模板来源区间 |
|---:|---|---:|---:|---:|
| 1 | DMR | -800 kHz | 0.800 s | 1.500～3.000 s |
| 2 | P25 | -400 kHz | 0.820 s | 6.000～7.500 s |
| 3 | dPMR | 0 kHz | 0.840 s | 0.200～1.700 s |
| 4 | NXDN | +400 kHz | 0.860 s | 1.000～2.000 s |
| 5 | TETRA DMO | +800 kHz | 0.880 s | 5.150～6.150 s |

这些输入区间都先在原始窄带文件上通过了对应制式的强证据确认和 PDU 解码。生成器把
每段模板重采样到 2.5 MS/s，并重复铺展为 5 秒活动区间；五路起点跨度为 80 ms，之后
增加 0.6 秒尾部。默认输出总长 6.48 秒、16,200,000 个复样点，CI16 文件为
64,800,000 字节（约 61.8 MiB）。

模板重复只服务于持续并发压力测试。模板边界可能产生相位不连续，且模板内部原有的短
静默仍可能形成多个 Epoch；JSON 清单明确记录源区间、重采样比、输出样点范围、随机
种子和幅度缩放，避免把它当作未经加工的真实空口数据。

## 3. 生成与运行

从工程根目录启动 MATLAB：

```matlab
startup
manifest = synthesizeFiveSignal2p5MHz('Overwrite', true);
```

默认生成：

```text
signal_data/synthesized_5sync_2.5MHz.rawiq
signal_data/synthesized_5sync_2.5MHz.rawiq.json
```

在图形前端中运行：

```matlab
app = radio_live_frontend( ...
    'DefaultFile', manifest.outputPath, ...
    'SampleRate', manifest.sampleRateHz, ...
    'CenterFrequencyHz', 0);
```

在频谱图上选择五个峰，或程序化加入频偏：

```matlab
app.StartPreview('StartTimer', false);
for offsetHz = [-800e3, -400e3, 0, 400e3, 800e3]
    app.AddOffsetHz(offsetHz, 'Refine', false);
end
app.RunIdentification();
```

## 4. 自动验收

真实数据验收没有加入 `tests.runAll`，因为它会创建约 62 MiB 文件、逐路执行五个真实
解码器并启动五进程池。手动运行：

```matlab
report = tests.runFiveSignal2p5MHzAcceptance;
```

默认步骤包括：

1. 重新生成确定性样例和 JSON 清单；
2. 检查文件长度、采样率、五路起点和频谱位置；
3. 每个载频只启用其预期制式，完成量化后 DDC 和 PDU 闭环；
4. 使用真实 `radio_live_frontend`、五个进程 worker、全部五制式竞争做 1× 回放；
5. 返回赢家、逐通道 PDU 数、最大输入滞后、最大解码队列和端到端实时因子。

默认 `StrictFrontend=false`，因为当前目的也包括记录并发失败，而不是遇到缺失通道就
丢失性能数据。需要把五路赢家和五路 PDU 作为硬门槛时使用：

```matlab
report = tests.runFiveSignal2p5MHzAcceptance( ...
    'Regenerate', false, ...
    'StrictFrontend', true);
```

只验证合成文件和逐路闭环、不启动实时前端：

```matlab
report = tests.runFiveSignal2p5MHzAcceptance( ...
    'Regenerate', false, ...
    'RunFrontend', false);
```

## 5. 2026-07-15 改造前基线

合成与逐路闭环稳定通过：

| 项目 | 结果 |
|---|---:|
| 五个实测谱峰 | -799.1 / -399.2 / +0.2 / +399.2 / +803.1 kHz |
| 各峰相对宽带噪声 | 约 63～68 dB |
| DMR 单路 | confirmed，19 PDU |
| P25 单路 | confirmed，27 PDU |
| dPMR 单路 | confirmed，13 PDU |
| NXDN 单路 | confirmed，101 PDU |
| TETRA 单路 | confirmed，177 PDU |

五阶段性能改造前，五路并发不是稳定实时。一次完整成功运行得到：

```text
winners = DMR / P25 / dPMR / NXDN / TETRA
PDU/channel = [28, 15, 13, 105, 96]
total PDU = 257
max input lag = 0.182 s
max decoder queue = 4.140 s
end-to-end RTF = 1.923
```

重复运行中也出现过 dPMR 未锁定、P25 已锁定但未在收尾前上报 PDU 的情况。对应运行的
最大输入滞后仍只有 0.071 秒，但解码队列达到 4.260 秒，端到端 RTF 为 1.769。因此：

- 频谱和文件输入仍能大致维持 1×，不能据此宣称解码实时；
- 五路在 80 ms 内出现时会一次产生最多 25 个制式 Probe，而进程池只有 5 个 worker；
- 五路 DDC 仍由主 MATLAB 线程逐路执行，12 ms 的每 tick 解码追赶预算不足以清空队列；
- EOF 时快速排空积压 IQ，异步 Probe 的完成顺序会影响 Epoch 结束前能否锁定和输出 PDU；
- 当前 2.5 MHz 三路基线仍可保留；五路近同步场景应作为后续全局任务调度和队列优化的
  压力回归，不应通过错开信号或延长探测窗口来掩盖失败。

## 6. 后续优化入口

优先级应为：

1. 建立跨载波的全局 Probe 调度器，避免每个通道各自一次提交五个任务；
2. 为快速制式先验或廉价特征设置优先级，同时保证各载波公平；
3. 将五路 DDC 扇出移出 UI 主线程，或实现共享本振/向量化/MEX 多通道抽取；
4. 把解码队列的增长率和每通道 backlog 作为显式运行指标；
5. 在严格验收中要求五路赢家、每路至少一条 PDU、无 overrun，并限制最大队列而不只看
   最终队列是否归零。

## 7. 五阶段改造及最终验收

2026-07-15 按上述压力基线完成了五个阶段，且每阶段先通过对应回归再进入下一阶段：

| 阶段 | 实现内容 | 主要效果 |
|---:|---|---|
| 1 | 每载波有界 `MaxInFlight`、公平 worker 配额、强赢家提前结束、所有 worker 协议入口预热 | 五载波时每载波最多占一个竞速任务，避免单载波一次占满进程池 |
| 2 | PSD 仅在显示周期计算最新帧、活动检测频谱噪声限频、基带 pending 预分配、UI 状态差异刷新 | 删除每个输入块上的重复 FFT、数组增长和无变化控件刷新 |
| 3 | 保守的 4FSK / π/4-DQPSK 候选门控，连续两次同家族才生效，并可在正在运行的 Race 中动态收窄 | TETRA 不再长期排在四个 4FSK Probe 之后；不确定时仍保留五制式 |
| 4 | 锁定解码器改为“仅传新 IQ + worker 侧有界历史”，设置最小推进量，并将 catch-up 环形缓冲裁成必要快照 | 降低 `parfeval` 序列化量和重复任务提交频率；仍保留整窗解码兼容语义 |
| 5 | 五路本振矩阵和单个矩阵 `DigitalDownConverter` 融合，固定本振表预计算 | 五路 DDC 从 6.232 ms/块降至 3.012 ms/块，约 2.07 倍；与独立 DDC 最大误差为 `2.62e-12` |

端到端复测又暴露并修正了三个调度问题：

1. `timer` 原先使用 `fixedSpacing`，会在每次回调完成后再等待一个 `Period`，把计算时间和
   20 ms 空等相加；现改为按墙钟周期调度的 `fixedRate`。
2. 已锁定的早期载波会持续补充 locked-decode 任务，可能让较晚的 TETRA Probe/catch-up
   饿死；现在分类和赢家追赶具有全局优先级，全部活动载波完成锁定后才恢复稳态解码。
3. 正常 1× 路径改为“读取一块、立即处理一块”，有界队列只服务于异步挂接和异常追赶；
   队列使用 head 索引常数时间出队。融合模式收尾也不再刷新五个未使用的独立 DDC。

严格验收命令：

```matlab
report = tests.runFiveSignal2p5MHzAcceptance( ...
    'Regenerate', false, ...
    'VerifyKnownChannels', false, ...
    'StrictFrontend', true, ...
    'Verbose', true);
```

最终一次完整严格运行得到：

```text
winners = DMR / P25 / dPMR / NXDN / TETRA
PDU/channel = [9, 4, 5, 16, 71]
total PDU = 105
input duration = 6.480 s
frontend run = 6.844 s
end-to-end RTF = 1.056
max input lag = 0.196 s
max decoder queue = 0.000 s
functionalPass = true
continuousDecodeRealtime = true
```

这里的 RTF 包含 100 ms 固定播放缓冲、Run 挂接和 EOF 收尾，不等同于纯 DSP
`处理时间 / 输入时长`；持续输入期间无解码积压，输入抖动保持在 250 ms 验收预算内。
关闭语义去重后，PDU 数会随异步窗口完成时刻变化，因此硬不变量是“五路赢家正确、每路
至少一条 PDU、无丢样/overrun、队列受控”，不是逐路 PDU 数必须与本次完全相同。

当前可以确认的是 **2.5 MS/s 离线文件、五个已知载频、五制式未知、MATLAB 软实时**。
这不证明 60 MHz 宽带发现已实时，也不等价于真实 SDR 生产队列。阶段 4 仍是有界历史
整窗重算适配器，而非五个协议均已实现持久符号/FEC 状态；约 30 秒的进程池、协议入口和
DDC 预备属于开始回放前的一次性冷启动成本，后续仍应继续缩短。

## 8. NXDN 真增量化后的复测说明

上述第 7 节记录的是五阶段改造完成时的一次完整严格通过结果。此后 NXDN 已改为跨块
保持 DSP、帧和链路状态的真正增量解码，并加入持久 worker 原型。NXDN 的重复整窗开销
显著下降，但重复压力运行显示 250 ms 最大输入滞后门槛仍有约 0.244～0.287 s 的波动，
最慢调用已经转移到 TETRA/其他旧协议兼容路径；五个制式赢家仍能正确识别。

所以第 7 节不能解释为确定性的硬实时承诺。当前停点的详细实现、测试证据与后续顺序见
[NXDN96 真正增量解码与持久 Worker 阶段记录](../recognition/NXDN96真正增量解码与持久Worker阶段记录.md)。

## 9. 四协议真增量化后的复测（2026-07-16）

在 NXDN 之后，DMR、P25 和 dPMR 也完成原生因果流式解码；五种制式的锁定任务全部
改为持久进程 Actor，只有 TETRA 内部仍执行有界历史整窗解码。相同 6.48 秒样例、不改
信号起点、不改 Probe 窗口，连续两次严格运行得到：

| 运行 | 赢家 | PDU/通道 | 最大输入滞后 | 端到端 RTF | 结果 |
|---|---|---|---:|---:|---|
| A | `DMR/P25/dPMR/NXDN/TETRA` | `[4 1 5 12 76]` | 0.238 s | 1.057 | 严格通过 |
| B | `DMR/P25/dPMR/NXDN/TETRA` | 断言前未汇总 | 0.307 s | 断言前未汇总 | 250 ms 失败 |

这说明五路功能闭环稳定，250 ms 输入滞后门槛仍受 TETRA、首次 Actor 启动、Probe/
Catch-up 与 MATLAB timer 长尾共同影响，不能稳定保证。运行 A 的冷准备耗时为
42.861 s，属于播放前进程池、Actor、协议入口和五路 DDC 预热，不计入 1x 输入 RTF，
但仍是后续必须优化的启动成本。详细实现见
[多协议真正增量解码与持久 Worker 第二阶段记录](../recognition/多协议真正增量解码与持久Worker第二阶段记录.md)。

## 10. 五协议全部真增量化后的复测（2026-07-16）

TETRA DMO 也已完成因果增量前端、slot/DMAC 和 session 状态化。五种已支持
模式在 `LOCKED` 后都只处理新 IQ，不再通过有界历史整窗重算来持续解码。
保持原 6.48 秒样例、五个频点、信号起点、Probe 窗口和 1× 回放速度不变，
连续三次严格运行得到：

| 运行 | 赢家 | PDU/通道 | 最大输入滞后 | 端到端 RTF | 结果 |
|---|---|---|---:|---:|---|
| A | `DMR/P25/dPMR/NXDN/TETRA` | `[4 1 5 18 71]` | 0.205 s | 1.048 | 严格通过 |
| B | `DMR/P25/dPMR/NXDN/TETRA` | `[5 1 6 16 76]` | 0.196 s | 1.047 | 严格通过 |
| C | `DMR/P25/dPMR/NXDN/TETRA` | 断言前未汇总 | 0.259 s | 断言前未汇总 | 250 ms 失败 |

A/B 的最终队列都为 0，冷准备约 41.3～41.6 s。C 仅超出门槛 9 ms，
最慢 Scanner Feed 为 0.1239 s，当时五路均已锁定，TETRA 通道占约
0.1062 s。额外拆分计时显示，1 s TETRA seed 的本地 CI16 压缩中位约为
0.6 ms，空闲单 worker 下含该 seed 的 Actor 接管中位约 19 ms。因此不应
缩短协议历史或改变样例来“修复”这个长尾；下一性能阶段应对 Probe、Catch-up、
Actor 握手/IPC、worker compute 和 UI timer 分别打点，并做数分钟循环回放的
backlog 斜率验收。

当前最精确的结论是：五路功能闭环和协议真增量化已完成，典型运行能满足
250 ms MATLAB 软实时预算，但尚未对长尾给出确定性保证。
