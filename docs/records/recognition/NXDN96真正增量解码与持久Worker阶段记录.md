# NXDN96 真正增量解码与持久 Worker 阶段记录

日期：2026-07-15

分支：`feature/streaming-protocol-race`

## 1. 本阶段目标

五阶段前端优化完成后，锁定制式仍通过“新 IQ + 有界历史整窗重算”兼容层持续解码。
这能减少传输量和无效调度，但不能消除重复滤波、重采样、同步和帧解析。本阶段选择
NXDN96 作为第一个真正增量化的协议，实现跨输入块保留 DSP、帧和链路层状态，并验证
持久 worker 内保存解码状态的可行性。

本记录是暂停开发时的检查点，不表示五个协议均已完成真正增量化，也不表示 MATLAB
五载波压力场景已经具备确定性硬实时保证。

## 2. 已完成实现

### 2.1 NXDN 帧与链路状态拆分

将原先一次性解码流程拆成可持续调用的状态接口：

- `nxdn.frameDecoderInit`：初始化帧同步、SACCH 和会话状态；
- `nxdn.frameDecoderFeedCandidate`：逐候选帧更新状态并产生新增 PDU；
- `nxdn.frameDecoderFinalize`：在输入结束时收尾；
- `nxdn.frameDecoderReport`：从当前状态生成报告。

离线 `nxdn.decode` 复用同一套帧解码状态，因此离线和流式路径不再维护两套独立的
链路层语义。

### 2.2 因果、跨块的 IQ 增量链路

新增：

- `nxdn.streamInit`
- `nxdn.streamDecodeChunk`
- `nxdn.streamFlush`

状态中持续保留 DC 去除、FIR/重采样、解调缓存、帧同步、会话及 SACCH 组合信息。
每次只消费新到 IQ，解调缓存有明确上限，不再对约一秒历史 IQ 反复执行整条解码链。

当前支持两类采样率路径：

- 78.125 kS/s 到 48 kS/s：使用状态化 `dsp.FIRRateConverter`；
- 120/125 kS/s 调谐输出：使用可跨块保持状态的 MATLAB 多相 FIR 路径。

### 2.3 持久 worker 原型

新增 `lockedDecoderActor*` 系列接口。进程 actor 在 worker 内创建输入队列并长期持有
协议解码器状态，客户端只提交新 IQ 并轮询轻量状态快照。为降低进程间序列化成本，
actor 输入支持 CI16 压缩，worker 收到后恢复复数样点。

当前默认启用范围有意保持保守：

- 只有 NXDN、且没有自定义 `DecodeFcn` 时进入持久 actor；
- 精确 120 kS/s 的 NXDN 可使用 `backgroundPool` 路径；
- 常见 125 kS/s 调谐输出仍使用进程 actor；
- DMR、P25、dPMR、TETRA 仍走有界历史兼容适配层。

## 3. 验证结果

`tests.runNxdn96Streaming` 对独立 NXDN96 样例完成了以下验证：

- 流式与离线结果一致：415 个 PDU、524 个有效帧、1062 个有效块；
- 两组不规则输入分块得到相同结果；
- 125 kS/s 多相重采样路径保持分块不变性；
- 已观察到的单路处理实时因子约为 0.16～0.31，具体数值受 MATLAB 预热状态影响。

`tests.runPersistentLockedDecoder` 验证了通用假 actor 和 120 kS/s NXDN 集成路径。
完整 `tests.runAll` 在本检查点通过，包括五制式样例、流式阶段、调谐 DDC、前端、
TETRA 与 NXDN 独立/流式回归。

## 4. 五载波压力场景的当前边界

NXDN actor 输入改为 CI16 后，一次诊断中 NXDN 提交与处理开销已降到约 1.2 ms，最慢
调用转移为 TETRA，单次约 118.7 ms。最近多次 2.5 MHz、五路近同步压力运行均能得到
DMR、P25、dPMR、NXDN、TETRA 五个正确赢家，但 250 ms 最大输入滞后门槛仍存在波动：
有运行约为 0.244 s，也有运行约为 0.287 s；端到端实时因子也会略高于 1。

因此当前准确结论是：

- NXDN 的重复整窗解码瓶颈已被实质性消除；
- 五路功能识别保持正确，但严格软实时门槛尚不能稳定保证；
- 当前主要瓶颈已经转移到 TETRA 和其他仍采用有界历史重算的旧协议路径；
- 早先一次五阶段严格验收通过仍然有效，但不能单凭该次结果宣称所有运行都满足
  250 ms 时限。

## 5. 暂停点与后续顺序

建议恢复开发后按以下顺序推进：

1. 将持久 worker 调度骨架扩展到其余锁定协议，先消除反复创建任务和大窗口传输；
2. 依次实现 P25、DMR、dPMR 的真正增量 DSP/同步/FEC 状态；
3. 单独分析并增量化 TETRA，其当前单次耗时和状态复杂度最高；
4. 进行长时间循环回放和 backlog 斜率测试，验收持续实时而不只是文件结束后追赶完成；
5. 在真实 SDR 的数据格式、块大小和时钟策略确定后，再接入正式生产者/消费者队列；
6. 若进入生产部署，保留 MATLAB 作为算法基准和回归平台，将实时数据面迁移到 C++。

每完成一个协议的增量化，都应同时保留离线等价性、任意分块一致性、内存有界性、
取消/收尾安全和多载波压力回归，避免以延长窗口或错开信号的方式降低测试难度。
