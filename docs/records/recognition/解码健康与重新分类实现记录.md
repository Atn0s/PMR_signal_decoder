# 解码健康与重新分类实现记录

## 1. 目的

一次协议确认不能代表整个输入永久保持同一制式。持续解码器为每个新窗口输出健康
状态，RaceCoordinator 用迟滞门限区分短暂校验失败、真正失锁和信号结束。

## 2. 初始门限

```matlab
cfg.lockedSuspectWindows = 3;
cfg.lockedLostWindows = 6;
```

门限按持续解码处理次数计数，当前默认 100 ms Chunk 下约对应 0.3 秒和 0.6 秒，但
整窗重算耗时和调用节奏可能改变实际墙钟时间。最终应改成绝对采样/时间门限，并用真实
衰落、碰撞和协议切换样本标定。

## 3. 状态迁移

```text
LOCKED
  -- 连续 suspect --> LOSS_PENDING
  -- 信号消失 -----> NO_SIGNAL

LOSS_PENDING
  -- 强证据恢复 ---> LOCKED
  -- 达到 lost ----> RECLASSIFYING
  -- 信号消失 -----> NO_SIGNAL
```

进入 `RECLASSIFYING` 时：

1. 保存 `previousProtocol`；
2. 清空当前 `selectedProtocol` 和 DecoderState；
3. 当前 Epoch 的 generation 加一；
4. 清空 ProbeState，下一 Chunk 重新启动五协议 Race；
5. 后续确认不同协议时产生 `PROTOCOL_SWITCH_CONFIRMED`。

信号能量已经消失时不会启动五协议计算，而是由活动检测直接关闭 Epoch。

## 4. 正确性约束

重新分类沿用同一 epochId，但必须使用新 generation。旧 Decoder Future、Catch-up
Future 或 Probe Future 即使稍后返回，也因 generation 不匹配被丢弃。

健康 `error` 与达到 `lost` 门限都能触发重新分类，但单次 `no_evidence` 不触发，避免
帧边界、短衰落或一个坏 CRC 造成协议抖动。
