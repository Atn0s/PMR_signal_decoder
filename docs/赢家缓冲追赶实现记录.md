# 赢家缓冲追赶实现记录

## 1. 目标

协议 Probe 读取环形缓冲快照，不能消费输入。确认制式时，实时采集位置通常已经超过
Probe 窗口末尾。阶段 5 负责从活动开始前的预触发位置重放赢家协议，并在追到当前
缓冲边沿后才进入 `LOCKED`。

## 2. 共用窗口解码

`radio.stream.decodeProtocolWindow` 是 Probe 和赢家重放共用的协议窗口解码入口：

```text
IqChunk
  -> 协议目标采样率
  -> 现有 frontend/decode
  -> PDU + diagnostics + frequency/timing
```

环形缓冲使用 complex single 控制内存；MATLAB R2022b 的 `resample` 只接受 double，
因此该入口在进入现有 DSP 前显式转换为 complex double。缓冲仍保持 single，不扩大
长期内存占用。

## 3. 追赶范围

```text
retained buffer start       candidate start       decode snapshot end
          |-----pre-trigger------|-----------------------|
```

`radio.stream.winnerCatchup` 按以下方式选择范围：

```matlab
desiredStart = candidateStart - preTriggerSamples;
catchupStart = max(ringBuffer.startSample, desiredStart);
catchupEnd   = ringBuffer.endSample;
```

若预触发部分已经被覆盖，`preTriggerTruncated=true`。若候选起点及其后续数据也不在
缓冲中，则追赶失败，不能假装已经完整解码。

## 4. 异步追赶和实时边沿

并行模式用 `winnerCatchupStart/Poll/Collect/Cancel` 在持久 process pool 上解码冻结的
缓冲副本，采集线程继续写入本地环形缓冲：

```text
启动重放时边沿 B ---- worker 解码 A..B ---- 完成时实时边沿 C
                                             B < C
```

如果结果只到 B，而当前缓冲已到 C，`RaceCoordinator` 以最新缓冲再启动一轮兼容
重放。只有 `catchupEndSample == current ringBuffer.endSample` 时才转入 `LOCKED`。
取消和结果接收继续检查 epoch/generation，旧 Epoch 的追赶不能写入新信号。

当前仍是整窗重算兼容层。输入速率长期高于整窗解码吞吐时，多轮重算可能追不上实时
边沿；阶段 6 必须用增量 DecoderState 取代它。

## 5. 绝对 PDU 位置

`radio.stream.stampStreamPdus` 为每个 PDU 增加：

```matlab
pdu.extra.stream = struct( ...
    'epoch_id', epochId, ...
    'source_sample', absoluteSourceSample, ...
    'source_time_sec', absoluteSourceSample / sourceFs, ...
    'window_start_source_sample', snapshotStart, ...
    'window_end_source_sample', snapshotEnd, ...
    'position_basis', basis);
```

窄带制式优先将 `extra.fs_start` 或 `extra.start_sample` 从协议目标采样率映射回源采样
率；TETRA 优先使用 `extra.start_time_s`。没有局部位置的会话汇总暂时使用窗口起点并
标注 `window_start_fallback`，后续增量解码应产生原生绝对位置，避免估算。

## 6. 协调器

`radio.stream.raceCoordinatorFeed` 把活动检测、异步 Race 和赢家追赶连接起来：

```text
NO_SIGNAL
  -> ACTIVITY_PENDING
  -> CLASSIFYING
  -> CATCHING_UP
  -> LOCKED
```

分支结果还包括 `AMBIGUOUS`、`UNKNOWN`、`ERROR` 和 `CATCHUP_ERROR`。输入丢样或
信号结束时会取消在途 Race/Catch-up 并清理 Epoch。

阶段 5 的 `LOCKED` 只表示已追到确认时的实时边沿；锁定后的持续逐 Chunk 解码和健康
监控属于阶段 6/7，尚不能把本阶段协调器直接视为最终 SDR 主循环。

## 7. 验证结果

测试覆盖：

1. 48 kHz 协议局部采样号到 78.125 kHz 源采样号的映射；
2. TETRA 时间位置到源采样号的映射；
3. 真实 DMR 从 `candidateStart-0.2 s` 重放并取得强校验 PDU；
4. 串行协调器完成 `CLASSIFYING → CATCHING_UP → LOCKED`；
5. 追赶期间新增 0.2 秒 IQ 后，第一轮结果正确显示落后，第二轮追到新边沿；
6. 信号结束后清理已锁定 Epoch。
