# 持续解码重叠适配层

## 1. 阶段定位

阶段 6a 在赢家追到实时边沿后继续处理新 Chunk，建立统一持续解码接口：

```text
DecoderState + RingBufferSnapshot
    -> NewPdus + Health + NewDecoderState
```

当前内部仍调用现有整窗解码器，并非最终的因果增量 DSP。该适配层先解决持续输出、
窗口边界、跨窗口去重和健康接口，之后可以逐协议替换内部实现而不修改协调器。

## 2. 有界重叠窗口

`lockedDecoderProcess` 只读取各协议最大 Probe 窗口长度：

| 制式 | 重叠回看 |
|---|---:|
| DMR | 1.5 s |
| P25 | 1.0 s |
| dPMR | 1.5 s |
| NXDN | 1.0 s |
| TETRA DMO | 6.0 s |

每次窗口终点是当前 ring buffer 边沿。窗口起点随输入前进，因此必须把协议局部位置先
映射成绝对源采样位置，才能识别相邻窗口中的同一 PDU。

## 3. 跨窗口账本

`streamPduKey` 组合协议语义键和 5 ms 绝对源采样桶：

```text
timed:<protocol semantic key>:<absolute source sample bucket>
```

CALL/SESSION 汇总使用 Epoch 内持久语义键，避免整窗解码器每次重建同一会话摘要。
普通定时键只保留当前回看范围内的记录，防止长时间接收导致账本无限增长。

这里没有调用现有离线 `radio.deduplicatePdus`，因为部分离线语义键会把同一窗口中
不同时间出现的相同控制 PDU 合并，从而使窗口前移后无法判断哪个实例是新事件。流式
去重策略仍属于后续优化项，需要结合实际产品输出粒度继续调整。

## 4. 健康输出

每次解码返回协议强证据结果：

- `healthy`：当前回看窗口有强校验；
- `no_evidence`：本次无强证据，尚未达到怀疑门限；
- `suspect`：连续 3 个处理窗口无强证据；
- `error`：前端或解码器异常。

状态中保存 `lastStrongEvidenceSample` 和 `consecutiveNoEvidence`。阶段 7 将用这些字段
驱动 `LOSS_PENDING` 和 `RECLASSIFYING`；阶段 6a 只产生健康信息，不擅自切换协议。

## 5. 协调器接入

`RaceCoordinator` 在 Catch-up 到达实时边沿时，用 Catch-up 已输出 PDU 初始化账本，
并把 `lastProcessedEndSample` 设置为 Catch-up 终点。之后每个新 Chunk 到达时调用持续
解码器，只向上游返回 `newPdus`。

信号结束或新 Epoch 创建会清理 DecoderState。丢样仍由 ChannelController 关闭旧
Epoch，不能跨不连续输入保留账本或健康状态。

## 6. 测试结果

测试包括一个可精确控制边界的合成解码器，以及 DMR、P25、dPMR、NXDN、TETRA
五个真实样本。验证内容：

1. 首窗均取得强确认；
2. 同一 ring end 重复调用输出 0 个新 PDU；
3. 追加 100 ms 后可继续解码；
4. 两次输出的 streaming key 不重复；
5. 协调器在 LOCKED 后会处理后续 Chunk；
6. 账本键随回看范围裁剪。

## 7. 尚未完成的真正增量状态

下一步需要把整窗重算逐步替换为：

```text
DDC NCO phase
causal FIR delay line
stateful resampler phase/history
FM discriminator previous IQ
symbol timing state
sync correlation tail
protocol frame/superframe assembler
session state
```

建议顺序仍为 NXDN/P25、dPMR/DMR、TETRA。替换过程中 `lockedDecoderProcess` 的输出
契约、绝对采样测试向量和账本测试保持不变。
