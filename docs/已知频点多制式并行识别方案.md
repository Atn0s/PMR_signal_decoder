# 已知频点多制式并行识别方案

状态：后续项目架构记录，当前不改变运行代码。

## 1. 背景与范围

实际应用通常已经给出中心频点，但不知道信号属于哪一种制式。当前已支持或正在
接入的制式包括 DMR、P25、dPMR、NXDN 和 TETRA，后续还会继续增加协议。

本方案只讨论：

```text
一个已知中心频点 -> 多个协议并行探测 -> 确认唯一制式 -> 完整解码
```

暂不讨论多个频点之间的并行扫描，也不把 TETRA 宽带盲扫列入本阶段目标。当前
开发重点仍是增加协议，并保持已有串行入口稳定。

## 2. 当前串行基线

当前 MATLAB 原生入口按下面的顺序执行：

```text
已知频点 IQ
-> 下变频到中心
-> 48 kHz 窄带分支：DMR -> P25 -> dPMR -> NXDN
-> 72 kHz windowed-IQ 分支：TETRA
-> 合并、后处理和去重
```

`radio.decodeIqEnabled` 会按协议注册顺序调用完整 frontend/decode。DMR 和 P25
共用 `c4fm_4fsk` frontend cache，dPMR 和 NXDN 使用各自独立 frontend，TETRA
使用独立的 72 kHz 扫描链路。

现状没有协议级 `parfor`/`parfeval`、赢家选择或任务取消。各解码器内部在找不到
同步、帧或校验结果时会返回空数组，但通常已经对输入窗口做过完整相关搜索，不能
等同于有时间上限的快速探测。

这条串行路径是后续并行项目的兼容性基线，在并行方案通过完整回归前必须保持为
默认执行方式。

## 3. 目标方案：并行探测，赢家完整解码

不要直接并行运行五个现有完整解码器。正确边界是将协议处理拆成两个阶段：

```text
probe   短窗口、低成本，寻找足以确认协议的证据
decode  只由确认匹配的协议继续执行完整解码
```

建议为 `ProtocolSpec` 增加可选探测能力：

```text
probeFcn
probeSampleRateHz
probeWindowSec
probeMaxDurationSec
decodeFromProbeFcn
```

统一探测结果至少包含：

```text
protocol
status            confirmed / no_evidence / rejected
score             统一置信度
evidence          同步、CRC、BCH、训练序列等协议证据
state             频偏、定时、同步位置及后续解码上下文
consumedSamples
```

协议确认不能只依赖一个相关峰。建议采用以下强证据：

| 制式 | 建议确认条件 |
|------|--------------|
| DMR | DMR 同步加至少一个通过协议校验的帧或链路控制块 |
| P25 | frame sync 加有效 BCH NID |
| dPMR | FS 同步加有效 CCH CRC/Hamming，低置信度时要求重复 |
| NXDN | FSW 加有效 LICH，并至少有一个 SACCH/FACCH/CAC/UDCH CRC 有效块 |
| TETRA | 有效训练序列加 SCH/S、SCH/H 或完整 DMAC-SYNC |

短窗口没有发现同步只能表示 `no_evidence`，不能立即判定制式不匹配。采集可能从
通话中间开始，不同协议也有不同的同步重复周期。因此需要按小窗口推进，超过该
协议的最大探测时限后才返回 `rejected`。

## 4. MATLAB 并行调度设计

协议竞速适合用 `parfeval`，不宜直接使用必须等待全部迭代完成的 `parfor`：

1. 已知频点只做一次 DDC。
2. 准备共享的 48 kHz 窄带探测输入和 TETRA 需要的 72 kHz 探测输入。
3. 向持久 parallel pool 提交每个协议的 `probeFcn`。
4. 使用 `fetchNext` 接收最先完成的探测结果。
5. 某协议达到强确认门限后，取消其他未完成任务。
6. 将探测阶段保存的同步、定时和协议状态交给赢家继续完整解码，避免从头计算。
7. 如果所有协议都超时而未确认，则返回 unknown/空结果，不强行选择制式。

概念流程：

```text
                   +-> DMR probe ---- rejected
已知频点短窗口 IQ +-> P25 probe ---- confirmed -> P25 full decode
                   +-> dPMR probe --- cancelled
                   +-> NXDN probe --- cancelled
                   +-> TETRA probe -- cancelled
```

DMR/P25 共用的前端应继续复用。第一版可以在协调端生成短窗口 frontend，再把结果
交给两个 probe；探测窗口较短时，进程间传输成本可控。parallel pool 应在扫描器
启动时建立并长期复用，不能为每个文件或窗口重复启动。

并行模式应作为显式选项加入，例如：

```matlab
'ExecutionMode', 'serial'         % 默认、稳定基线
'ExecutionMode', 'parallel-race'  % 后续分支实现
```

## 5. MATLAB 与 Python 选型评估

| 维度 | MATLAB | Python |
|------|--------|--------|
| 当前协议完整度 | DMR/P25/dPMR 原生链路已迁移，TETRA 只在 MATLAB 侧实现 | 离线 registry 只有 DMR/P25/dPMR，没有 TETRA |
| 当前主线位置 | 新协议和 TETRA 工作都在 MATLAB 项目继续推进 | 作为兼容后端和 golden baseline |
| 并行基础 | 尚未实现协议竞速，但可使用 Parallel Computing Toolbox 的 `parfeval` | `realtime/` 已有 `multiprocessing.Pool` worker，但按频点/窗口派发，不是协议赢家竞速 |
| 开发代价 | 可直接复用现有五制式和测试 | 需要先回迁 TETRA/NXDN 及后续新增协议，并长期维护两套一致性 |
| 部署能力 | 依赖 MATLAB 环境和可能的 Parallel Computing Toolbox 许可 | 更适合无 MATLAB 许可、长期服务化、容器化和 SDR 实时部署 |

阶段性决策：**并行协议识别继续放在 MATLAB 中实现。**

原因是当前目标是继续纳入更多制式，而协议实现和 TETRA 能力集中在 MATLAB 侧。
现在切回 Python 会先产生协议回迁、结果对齐和双端维护成本，延迟实际功能建设。
Python 已有的多进程实时框架值得保留为将来的部署参考，但不能抵消当前协议能力
缺口。

出现以下条件时，再重新评估是否将调度层或完整系统迁回 Python：

- 最终产品必须脱离 MATLAB 许可独立部署；
- 目标从离线/批处理转为长期运行的 SDR 实时服务；
- Python 已具备与 MATLAB 相同的协议集合、TETRA 能力和 golden parity；
- MATLAB worker 启动、数据复制或内存成本经过基准测试后不能满足时延目标；
- 需要容器化、远程服务编排或跨平台边缘部署。

## 6. 分支和稳定性策略

后续项目建议从完成并稳定提交的 scanner/TETRA 基线新开分支，例如：

```text
feature/known-frequency-protocol-race
```

开发约束：

1. `serial` 始终保持默认，现有调用不改变语义。
2. 新协议先接入串行 registry、样本和回归，再实现可选 `probeFcn`。
3. 并行调度只通过新增 execution mode 启用，不直接改写稳定串行函数。
4. 串行和并行模式必须在同一批样本上输出相同的协议和关键 PDU。
5. 必须记录 time-to-confirm、完整耗时、CPU、内存、误判和漏判。
6. 首个版本只支持已知中心频点，不顺带扩展多频点并行或 TETRA 盲扫。

建议实施顺序：

```text
定义 probe/result/state 接口
-> 为四个协议实现串行 probe 并验证确认条件
-> 增加 protocol-race 调度器但仍串行执行 probe
-> 替换为 parfeval 并加入取消机制
-> 增加 serial/parallel parity 和性能回归
-> 评估是否默认启用并行模式
```

先验证探测语义，再加入并行，可以避免把现有完整解码函数直接并行后仍然无法
快速失败或安全取消。

## 7. 当前 blind_search 行为

当前 MATLAB 工作区中，当 `BlindSearch=true`、`ProtocolNames={}` 且没有
`FreqList` 时：

1. `resolveScanProtocols` 选择 `defaultBlindSearch=true` 的协议；
2. 当前 DMR、P25、dPMR、NXDN 满足该条件；
3. `radio.psdBlindSearch` 先按 4.8 kHz 调制带宽聚合 PSD 能量，再生成信道级候选，
   避免把同一 4FSK 信号的多条调制谱线误判为多个载波；
4. 每个候选频点按 DMR、P25、dPMR、NXDN 串行尝试；
5. TETRA 不参与，也不会因为未指定协议而报错；
6. 没有候选峰或没有任何协议成功时，最终返回空 PDU 数组。

如果显式指定 TETRA，同时要求无 `FreqList` 的宽带盲扫，则当前会抛出
`radio:scanIq:TetraBlindSearchUnsupported`。

另一个需要保留在测试中的配置边界是：当前 `resolveScanProtocols` 优先判断
`BlindSearch`。因此同时设置非空 `FreqList`、`BlindSearch=true` 且不指定协议时，
协议列表仍按 blind-search 默认值解析，TETRA 会被排除。对于已知频点的五制式
串行扫描，应明确设置：

```matlab
PROTOCOLS = {};
FREQ_LIST = [knownOffsetHz];
BLIND_SEARCH = false;
```

这时当前工作区会尝试 DMR、P25、dPMR、NXDN 和 TETRA。并行项目不应顺带改变上述
blind-search 语义；如果未来要修改，应作为独立需求、独立测试和独立提交处理。

## 8. 验收标准

- 五种已知样本都能由正确协议确认，其他 probe 在时限内退出或被取消；
- 强校验之前不宣布赢家，噪声和未知制式不被强行分类；
- 串行模式输出和当前 golden/smoke baseline 一致；
- 并行模式和串行模式输出的协议及关键 PDU 一致；
- 增加新协议只需注册 spec/probe，不修改调度器核心；
- 并行模式的 time-to-confirm 明显优于完整串行遍历；
- 当前无制式 blind-search 运行 DMR/P25/dPMR/NXDN，TETRA 仍不参与。
