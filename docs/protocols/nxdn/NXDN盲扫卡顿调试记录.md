# NXDN盲扫卡顿调试记录

日期：2026-07-10

## 1. 现象

使用 `scanner.m` 扫描 `nxdn96_2_78125.rawiq`，配置如下：

```matlab
PROTOCOLS = {};
FREQ_LIST = [];
BLIND_SEARCH = true;
```

MATLAB长时间没有输出结果，表现为扫描过程卡住。测试文件采样率为
78,125 Hz，包含约58.25秒的单路NXDN96信号。

## 2. 根因

旧版 `radio.psdBlindSearch` 直接在原始Welch PSD上寻找局部峰值。NXDN96是
4FSK信号，同一个载波会形成多条明显的调制谱线。旧算法把这些谱线识别成
34个独立候选频点，范围约为 -7.17 kHz 到 +7.69 kHz。

盲扫对每个候选分别执行下变频、重采样，并串行运行DMR、P25、dPMR、NXDN
四个完整解码器。因此同一条58秒录音被重复处理34次。单候选四制式处理约需
20秒，旧路径的总耗时可能达到十余分钟。这不是死循环，而是候选数量错误造成
的计算量放大，同时旧入口没有阶段进度输出。

## 3. 修复

`radio.psdBlindSearch` 现在先在4.8 kHz调制带宽内聚合线性PSD能量，再在聚合
结果上寻找局部峰值，并使用5 kHz非极大值抑制间距。这样同一4FSK载波的谱线
会形成一个信道级候选，而中心间隔为6.25 kHz及以上的相邻标准信道仍可分别
保留。

边缘频点的聚合使用有效样本数归一化，避免卷积补零降低频谱边缘候选的能量。
`psdMaxCandidates` 默认保持为 `inf`，不通过硬数量上限静默丢弃宽带中的较弱
候选；需要限制资源时可以由调用方显式设置。

扫描入口增加可选 `ShowProgress` 参数，`scanner.m` 默认启用，运行时会输出候选
数量、当前候选序号、频偏和该候选的PDU数量。库函数默认仍保持安静。

## 4. 修复效果

在完整 `nxdn96_2_78125.rawiq` 上，保持 `PROTOCOLS={}`，即继续串行尝试
DMR、P25、dPMR和NXDN：

| 指标 | 修复前 | 修复后 |
|---|---:|---:|
| PSD候选数量 | 34 | 1 |
| 候选中心 | 多个调制谱线 | 约 +0.2 kHz |
| 端到端耗时 | 预计十余分钟 | 约22.6秒 |
| 去重后PDU | 未在合理时间完成 | 16，全部为NXDN |

耗时数据会随MATLAB版本、CPU和绘图配置变化，候选数量和解码结果是本次回归的
主要稳定指标。

## 5. 对计算量和漏检的影响

PSD聚合只作用于长度为4096的Welch频谱，增加的卷积计算相对于整段重采样和四套
协议解码可以忽略。正常情况下候选数量明显减少，因此总CPU占用和耗时下降，
不会增加完整解码器的调用次数。

这项修改不能保证任何信噪比下都绝不漏检，因为盲扫仍依赖“噪声中位数+15 dB”
门限。极弱信号、占满大部分输入带宽的密集信道、中心间隔小于5 kHz的非标准
信号仍可能被遗漏或合并。这些是PSD候选搜索的能力边界。对于项目当前支持的
标准窄带制式，最小关注中心间隔为6.25 kHz；合成回归已验证两条相隔6.25 kHz
的窄带4FSK信号会产生两个候选。

对于已知中心频点的实际应用，应继续优先设置 `FREQ_LIST` 并关闭
`BLIND_SEARCH`。该路径绕过PSD候选搜索，因此完全不受本次候选聚合参数影响。

## 6. 验证

- 真实NXDN96样本：单信道只生成一个候选，NXDN PDU正常解出；
- 合成单路4FSK：多调制谱线只生成一个候选；
- 合成相邻信道：中心相隔6.25 kHz的两路窄带4FSK分别生成候选；
- NXDN96专项测试通过；
- DMR、P25、dPMR、NXDN、TETRA全套MATLAB回归通过；
- MATLAB `checkcode` 和 `git diff --check` 通过。
